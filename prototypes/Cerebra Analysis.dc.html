01 — Product Vision

Cerebra is defined as a personal learning operating system — a tool whose purpose is long-term concept mastery and retention, not content consumption or course completion. This positions it firmly against LMS tools, note-taking apps, and productivity dashboards.

"The system should become a living representation of the learner's knowledge state rather than a repository of notes."

— Cerebra Technical Architecture, p.18

The north-star user question the system must answer at any moment:

What do I know?
Mastery & heat state per concept
What am I forgetting?
Retention score & heat decay
Why am I struggling?
Dependency engine & root weakness detection
What should I study next?
SM-2 scheduling & learning paths
How are concepts connected?
Knowledge graph with prerequisite/dependency edges

02 — Design System

Philosophy

The design vocabulary is anchored in a single metaphor: Earthy Glass — the warmth of analogue study tools (leather notebooks, botanical journals, wooden desks) married to the translucency of modern spatial interfaces. The interface should feel like a personal study sanctuary, not a productivity cockpit.

Surface System

Four elevation levels create depth without drama. The glass specification (20px blur, 12% white, subtle border) is precise and must be applied consistently:

Level Name Used For Properties
0 Background App canvas Cream paper. No borders.
1 Paper Card Notes, content sections Slight elevation. Soft border.
2 Glass Panel Widgets, concept summaries blur:20px · rgba(255,255,255,0.12)
3 Floating Surface Side panels, graph inspectors Elevated. Soft shadow.
4 Modal Review dialogs. Use sparingly. Highest elevation.

Motion System

Motion is functional, not decorative. Three durations cover all cases. Graph animations should feel organic — nodes gently reposition with no sudden jumps. Bounce and elastic effects are explicitly forbidden.

Domain Model

The architecture enforces a strict hierarchy: Subjects → Concepts → Events → Metrics. The most important design decision is the separation of events as truth from metrics as projections. Mastery scores are never stored on the concept — they are always recomputed from evidence.

Subjects
Independent domains: Linear Algebra, DSP, Probability…
Concepts
First-class entities with title, intuition, formal definition, notes. No mastery stored here.
Events (source of truth)
recall_events · problem_attempts · explanation_events — immutable log
Metrics (projections)
concept_metrics computed from events. Rebuildable. Not authoritative.

Mastery Formula

mastery_score = (recall_score × 0.50)
               + (problem_score × 0.30)
               + (explanation_score × 0.20)
50% Recall performance
30% Problem solving
20% Explanation quality (Feynman)

Normalized to 0–100. Retention score separately models the forgetting curve, outputting a probability (0–100) of successful recall at review time.

04 — Cross-Document Observations

Strong Alignments

Heat states bridge design and data
The color system (Forest → Moss → Sand → Clay → Muted Charcoal) maps exactly to the five heat states derived from retention scores (90+ → 75–89 → 50–74 → 25–49 → 0–24). Design and data model are unified — no translation layer needed.
Concept Page is the priority on both sides
The design doc names it "the most important page in the system — design priority: Highest." The architecture doc gives it the most detailed component breakdown. Both teams are aligned on where the most UX investment should go.
Graph layout is intentionally decoupled
The graph_layouts table separates visualization state from knowledge relationships — a mature decision that enables the design's "explorable atlas" feel without coupling visual positions to semantic data.
Events-as-truth supports non-gamified design
Because mastery cannot be manually edited and is computed purely from evidence, the system structurally prevents the gamification patterns the design doc explicitly rejects (streaks manipulation, score padding, social competition).

Tensions & Open Questions

TailwindCSS vs. the spacing system
The design system defines a strict 4px-base spacing scale (4, 8, 12, 16, 24, 32, 48, 64). Tailwind's default scale is also 4px-based but uses arbitrary class names — the team needs a Tailwind config that maps spacing and colors to the Cerebra palette explicitly, or visual drift will occur at implementation.
Glass surfaces require a non-solid background
The glassmorphism spec (backdrop-filter: blur 20px) only has visual effect when layered over non-uniform backgrounds. The design's Cream (#F5F1E8) base is nearly uniform, so glass panels may appear flat unless the background carries texture, imagery, or the Knowledge Graph canvas.
Phase 6 defers the designed interface to last
The MVP roadmap places the Earthy Glass visual polish in Phase 6. Phases 1–5 presumably use a utilitarian UI. This is pragmatic but creates a risk: interaction patterns built without the glassmorphism context may need re-layout when Phase 6 arrives. Partial glass treatment should be introduced earlier.
Review interface assessment buttons need visual design
The design doc says assessment buttons (Forgot / Partial / Mostly Correct / Perfect) "should be visually distinct" but does not define how. These map to SM-2 score values 0–3. Their visual treatment is critical UX — color, size, and layout will influence recall ratings and therefore mastery scores.
No authentication or multi-user model defined
All entities use UUID primary keys and there is no user_id in the schema. This is consistent with the "personal" framing but means collaborative features or auth will require a schema migration. Worth confirming whether single-user is a long-term constraint.
